Task: Identify and Outline Requirements
This task describes how to find and outline the functional requirements for the system so that the scope of work may be determined for the proposal.
Disciplines: Technical Requirements Development
Purpose
The purpose of this task is to identify and capture functional requirements for the proposed system. These requirements form the basis of communication and agreement between the stakeholders and the development team on what the system must do to satisfy stakeholder needs. The goal is to understand the requirements at a high-level so that the initial scope of work can be determined.  Further analysis will be performed to detail these requirements prior to implementation. This task does not ordinarily include identification of non-functional requirements, which are covered in a separate task.

        Main Description

        This task will identify and capture functional and other system wide requirements.  The intent is to capture a relatively small but complete set of requirements than define the system scope and behaviors necessary to address the client’s needs.  The requirements should cover

        ·         What the solution does in terms of business, management, audit, workflow, and/or administrative functionality

        ·         What the solution produces in terms of screens, reports, messages for other systems, and/or physical effects.

        ·         How the solution operates, in terms of use cases or epics, roles (who has what abilities to affect system behavior), states and modes, and interfaces.

        Additional requirements areas can be found in the Guideline: Developing Requirements Matrix and the Checklist: Requirements Taxonomy Checklist. 

        A structured approach to the identification and documentation of requirements includes four discrete steps:

        1.      Understanding what resources (time, documentation, and stakeholders) are available to support the task

        2.      Plan the appropriate requirements elicitation and identification techniques necessary to exploit the resources identified

        3.      Executing the requirements identification and elicitation plan

        4.      Verifying, validating, and publishing the requirements

        The plan in the second step is a tailored collection of requirements identification or elicitation techniques.  The two most common approaches for identifying and eliciting requirements are 1) reviewing existing documentation and 2) working directly with client stakeholders.  Most often both approaches are used.  Existing documentation can include the inputs associated with this task (Project Definition, Opportunity Plan, etc) and any client published information relevant to the proposal (such as a Request for Proposal or equivalent). 

        Stakeholder collaboration is best executed with a structured approach.  As such, when interacting with stakeholders, the selection of the form of stakeholder interaction and the specific topics to be addressed in that forum should be pre-planned, as defined in the steps below.

        As part of the stakeholder interaction, this task will most likely develop a limited number of initial use cases at a conceptual level and identify additional functional requirements. Developing initial use cases involves identifying actors and defining selected use cases with primary scenarios.  Use cases are descriptions of how the users (actors) interact with the proposed system. These use cases are intended to provide initial insights into system functional requirements and are a more specific description of functional requirements than the content of the Project Description. They may also be used as a basis for basic prototyping activity.

        The last step in this task is the publication of an agreed to set of requirements.

        Key Considerations

        Sometimes, there are only a few functional requirements and capturing them in the Project Definition is sufficient. In this case, Requirements Matrix which is one of the outputs of this task will not be needed. It may still be necessary to create the Use Case Model. If there are minimal non-functional requirements, it may be appropriate to combine them with functional requirements in the Requirements Matrix.

        EnterpriseArchitecture Influence: Existing enterprise architecture is an important influence on solution design. For additional guidance on this topic, see the associated Guideline: The Role of Enterprise Architecture in Solution Design. The following EA artifacts (or their equivalents) should be considered when performing this task (developing the Use Case Model).

        • Business Event List (BUS 101) Business events are a potential source of information about the type and nature of use cases that should be modeled.
        • Business Roles and Locations (ARC 303) Understand and define the roles and user groups
        • Process Identification (BUS 312) Process Identification provides source of information that may be helpful in identifying tasks and activities within the Use Case Model.
        • User Groups (ARC 311) User Groups provide source of information that may be helpful in identifying actors within the Use Case Model.

        Agile Development: In some situations, it may be known that implementation of the proposed solution will be accomplished using Agile Development. Although Use Cases may be used with Agile Development, User Stories provide an alternate but not equivalent means of gathering high level requirements. User stories are documented according to the artifact: User Story (ART 0650), typically using the following format. As a [user role], I want to [goal], so I can [reason]. Although User Stories and Use Cases are not commonly combined within a single agile development project, the typical User Story format may be helpful as a style for documenting certain requirements derived from user interviews even if Use Cases will be the primary approach used in pre-sale solution design.

        If Design Thinking has been used in the Ideation Task to define the Hills for a business initiative, then the Project Definition should have the directive on which Hills are to be included in the proposal to the client. Be sure to reference those Hills specifically in the Requirements work products in this task. Do not consider the Hills to be a sufficient statement of requirements as they are related to key business outcomes from the stakeholder for this initiative and they will inform the further elaboration of requirements in User Stories if Design Thinking and Agile techniques are adopted for the delivery project.
