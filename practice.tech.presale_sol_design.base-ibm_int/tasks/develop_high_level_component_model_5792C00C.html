Task: Develop High Level Component Model
This task will develop a high level component model in a pre-sale (Team Solution Design) context in order to describe the high-level structure of the system describe the responsibilities, relationships, boundaries and interactions of components.
Disciplines: Technical Architecture
Purpose

The purpose of this task is to develop a high level component model in a pre-sale (Team Solution Design) context in order to:  

  • describe the high-level structure of the system
  • describe the responsibilities, relationships, boundaries and interactions of components
  • document how application/technical parts of the system are related
  • specify how existing, acquired and developed components are related
  • provide a definition of the components that have to be placed on to the operational model

        Main Description

        This task will help identify important information about components for the proposed system by creating a component model. A component is a modular unit of functionality, which makes this functionality available through an interface. A component is viewed broadly and is literally any piece of software that has an interface and makes sense to portray as a component. Components may be made up of other components. They may be large or small. As general guidance, at a high level, components depict the major subsystems and boundaries (interfaces) of the overall system. Later they can be decomposed or refactored as necessary. Components are often built or acquired as a cohesive unit, separate in some way from other components. Components are described by responsibilities (based on functional requirements), required service levels, performance, capacity and availability.

        Steps
        Identify High Level System Components

        Identify the primary components of the proposed system. The Architecture overview diagram is an ideal source to identify candidate components. The System Context will also imply the presence of components and interfaces. Initially, the definition of a component can be considered at a logical level. Later stages of design will refine the component model and progress toward physical views.


        Describe Components
        Develop a component description for each component. This description will describe component responsibilities, required service levels, component users, performance, capacity and availability requirements and potential sources (make / buy). When appropriate, include the subject areas (data) that the component is responsible for.
        Define Component Relationships
        Describe key component relationships using a component relationship diagram. This will help identify important interfaces. Typical component relationships indicate a using (dependency) relationship, such that one component uses the interface offered by another component. Other arbitrary types of dependencies, such as implementation sequence may also depicted described
        Describe Component Interactions

        Develop a component interaction diagram to describe key component interactions. This will help explain how the proposed system operates.

        Key Considerations

        Enterprise Architecture Influence: Existing enterprise architecture is an important influence on solution design. For additional guidance on this topic, see the associated Guideline: The Role of Enterprise Architecture in Solution Design. The following EA artifacts (or their equivalents) should be considered when performing this task.

        • Application Function Model (ARC 302) The Application Function Model is helpful in understanding what functions are necessary for a solution and how current or proposed applications fulfill the required function.

        • Component Business Model (Architecture View) (ARC 402) This more detailed view of the Component Business Model can provide substantial insight into component interactions for those IT components supporting automated aspects of business activities and business component interactions.

        • Data Stores (ARC 306) The Data Stores work product is helpful in understanding which business information is relevant for a solution and which current or proposed applications (should) own and provide access to certain groups of data.

        • Enterprise Technology Framework (ARC 308) The ETF provides guidance in selection of components (including any relevant technology / product standards) based on pre-established architectural building blocks (ABBs).
