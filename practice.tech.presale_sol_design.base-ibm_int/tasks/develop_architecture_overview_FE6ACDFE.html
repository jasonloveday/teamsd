Task: Develop Architecture Overview
This task will high level abstraction of the proposed system or application as a basis for discussion, explanation and evolution of the architecture.
Disciplines: Technical Architecture
Purpose
To convey to the client the high level abstraction of the proposed system or application. The diagram will serve as a basis for discussion, explanation and evolution of the architecture. The diagram may also serve as a starting point for a services engagement.

        Main Description

        The architecture overview is key to communication with the client and acts as summary of the solution. It should be used as a basis for agreement on the fundamental content of the proposed system or application. To create the architecture overview, the IT Architect would determine some or all of the following and reflect the findings in the overview diagram.

        • What non-functional requirements exist?
        • What system constraints or preferences exist? (These are typically the result of prior decisions that affect this project)
        • What are the goals for this project?
        • What application requirements have been defined?
        • What are the system or application boundaries?
        • What systems or subsystems are known to be a part of this system or application?

        The architecture overview diagram is rather high level. It is something that could be understood by anyone ranging from a client architect or designer to a client executive. Format of the diagram is flexible and informal. When possible, reuse existing reference architectures, modifying as necessary.  Examples of content would include the following:

        • Major application systems or subsystems
        • Data stores
        • Client delivery channels such as VRU, internet
        • Client actors such as loan officer or CSR
        • High level message or data flows
        • Important boundaries that will affect more detailed design

        A text description should accompany the diagram to explain important aspects of the overview that are not obvious from the diagram. The point of the overview is to convey the highest level essence of the solution, unburdened by details such as choice of products, interface complexity etc. Additional details will be contained in other work products, particularly the high level operational model.
        Steps
        Analyze Input Artifacts

        Review and analyze all existing and relevant artifacts that will help in understanding the requirements. In particular look at the objectives, constraints, and measures of success for by reading relevant customer documents, including artifacts that describe project goals. The following are examples of information that may be discovered.

        • Identify all external systems that must connect with or be shared with the new system.  If a System Context is available, it will contain this information. Examples would include existing client systems such as VRU's or workstations, legacy systems, databases etc. If a system context diagram is not complete, use this information as input for it.
        • Identify classes of users, types of workstations and delivery or access channels.
        • Identify any constraints on the architecture. These can be found in the Non-Functional work product if available. An example would be that connections to a particular legacy system must connect via MQ.
        • Identify major high level application components or sub-systems and how they relate to each other.
        • Identify major nodes that would be used in the system. These typically equate to client or server platforms on which application software components and subsystems operate.
        • Identify placement of components and subsystems on nodes. This may be very approximate initially.
        • Identify geographical boundaries that affect placement of nodes.
        • Identify all connections between nodes and between components.
        • Identify technology and architectural alternatives that could be used to develop the architectural overview. Initially it may be helpful to develop multiple diagrams to illustrate alternatives that should be considered. Some of these alternatives may have already been determined and reflected in the Architectural Decisionss.
        • Identify primitive use cases that illustrate basic system behavior. These can be used to test the architecture overview and as an aid in explaining the architecture. (They may already exist or may need to be created and reflected in the Use Case Model)
        • Identify any reference architecture that may be used as a starting point.

        Envision Alternate Architectures
        Using the requirements and list of potential assets and products create one or more potential architectures. Envision possibilities for the architecture by considering the nature of what is being requested and the kind of architecture that will be appropriate. Draw on experience to hypothesize about system concepts to find analogies, and investigate industry models and other model assets.
        Evaluate Alternative Architectures
        Evaluate the alternative architectures and assess the pros and cons of each. Document the rationale for all decisions as Architectural Decisions. This may require experimentation, benchmarking and prototyping. Assess the viability of implementing the proposed architecture.  If appropriate, build risk identification and mitigation prototypes. Review the alternatives with the sponsors and come to a decision.
        Document Selected Architecture
        Document the selected architecture using an appropriate drawing or modeling tool.

        Key Considerations
        Although this task indicates Candidate Asset List as an input, it is likely that one would first begin to develop a very preliminary architecture overview (a sketch) in order to inform the search for candidate assets. Candidate assets may in turn be helpful in developing a more detailed architecture overview.
