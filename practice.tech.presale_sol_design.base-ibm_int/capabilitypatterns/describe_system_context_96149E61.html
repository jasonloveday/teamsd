Task: Describe System Context
This task will describe the system context to support additional analysis and better understanding of the proposed system.
Purpose

In the pre-sale environment, describing the system context is essential for understanding the system to be proposed. This task will discover and define how the proposed system will interoperate with other existing systems. It will also establish boundaries on the scope of the proposed system. The output diagram can highlight interface requirements necessary to quickly establish feasibility of using a particular technology to implement the proposed system.

Main Description
In this task, the proposed system is treated as a black box (the "target system") with connections to users and other systems. The System Context diagram is the resulting work product. This diagram documents all connections between the proposed system (target system) and external users and systems/components. Associated with each connection are various attributes such as data description, protocol, formats, frequency, volume, model of interaction (synchronous, asynchronous), etc. This context will constrain the proposed system with regard to the interfaces that must be implemented. The System Context may be developed as a free form diagram or as a more formal set of views. These views are based on four viewpoints which reflect various combinations of Functional, Operational, physical and logical. The associated guidance explains the modeling approach for these more formal views.
Steps
Establish System Scope and Boundaries

Establish the scope and boundaries of the target system by determining what will be included in the system and what external systems or actors will interact with the new system. Represent the planned (target) system as a simple circle with external systems and actors represented outside of the circle.


Document External Systems Relationships

Identify all interfaces on the boundary which connect some aspect of the proposed (target) system with external systems and actors. Identify Information regarding events, formats, protocols, etc. of the boundary interfaces.

Complete System Context Diagram

Draw the free form system context diagram, showing:

  • proposed (target) system
  • external systems and actors
  • relationships / interfaces
  • relationship / interface characteristics at a detail appropriate for pre-sale context

As an alternative, or in addition to the free form diagram, consider developing relevant formal views. These may include the following.

  • Logical Functional view: Reasoning about what the system does for or to its actors
  • Physical Functional view: Reasoning about the users and IT systems it connects to
  • Logical Operational view: Reasoning about the nature of the access mechanisms used by its actors
  • Physical Operational view: Reasoning about the connections needed with its users and other IT systems

For guidance on developing these formal views, consult the associated guidance.
 

Key Considerations

Enterprise Architecture Influence: Existing enterprise architecture is an important influence on solution design. For additional guidance on this topic, see the associated Guideline: The Role of Enterprise Architecture in Solution Design. The following EA artifacts (or their equivalents) should be considered when performing this task.

  • Application Function Model (ARC 302) The Application Function Model provides context for a solution by outlining the functional boundaries (which business function are we changing) and identifying relevant external systems (system actors) to interface with

  • Business Event List (BUS 101) External business events are a potential source of information about responsibilities of the target system and requirements involving external agents, access mechanisms and exchanges.

  • User Groups (ARC 311) User groups can provide information regarding external agents modeled as Users.
