Task: Develop High Level Operational Model
This task will develop a high level model in a pre-sale (Team Solution Design) context in order to provide a preliminary view of the system for use in understanding customer requirements and general architectural approaches.
Purpose

The purpose of this task is to develop a high level  model in a pre-sale (Team Solution Design) context in order to:  

  • Provide a preliminary view of the system for use in understanding customer requirements and general architectural approaches.
  • Gain understanding of major elements of the IT system to be built such as primary system nodes, connections, locations, major components (including technical infrastructure) and existing assets to be reused.
  • Gain an understanding of customer preferences and constraints.
  • Establish a mechanism for early discussion regarding functional characteristics of the system by enabling basic design walkthroughs.
  • Enable early analysis of non-functional requirements such as performance, availability and capacity, including confirmation of the viability of a solution through specification of the expected non-functional characteristics of nodes and components.
  • Enable the evaluation and selection of products that could be used in the proposed system.
  • Enable early estimates of the cost of the infrastructure to be used both for budgeting and as part of the business case for the solution.
  • Provide a basis for evaluating and selecting products that could be used in the proposed system.

Main Description

This task will help identify important information about the operational aspects for the proposed system by creating a high level Operational Model. The Operational Model at a physical level is a representation of a network of computer systems, nodes and their associated peripherals and the systems software, middleware and application software that they run. 

This representation must explicitly consider where such nodes, components and services are provided by a Cloud Service Provider. Cloud Computing does not remove the requirement to develop an Operational Model, but it can provide a combination of acceleration, constraints and complexity to the production of the work product:

  • Acceleration because existing patterns of deployment and service provisioning for infrastructure and application components can be reused. These patterns may be accessible within the cloud provisioning environment or via reference models.
  • Constraints as the cloud service provider provides a given set of components and accompanying service level agreements ( with implicit architecture decisions for component selection) which become boundary conditions for the non-functional requirements which your Solution Design can support.
  • Complexity where the operational model for a Solution Design must span various domains for a hybrid cloud solution with integration of heterogeneous components, services and service levels.

The operational model for a pre-sale context will be more likely developed at the logical level with considerably less detail. The high level Operational Model will contain some or all of the following model elements:

  • A Logical Operational View that shows the topology and geographic distribution of the system, the definition of the nodes (computer platforms) and network connections, and where and how users and external systems interact with the system being developed.
  • A set of inter-node descriptions. Each description will include a node description, applicable non-functional requirements, and an inventory of components (grouped as a deployment unit) that will be deployed on this node.
  • Description of connections arranged as a table or matrix.
  • For more complex systems, a table that depicts how deployment units (groups of components) map to specific nodes.
  • A walkthrough description that illustrates typical system function for one or more key use cases.
  • If appropriate, a description of the systems management strategy, including decisions about centralized vs. distributed managing stations, backup and recovery strategy, software distribution models and approach, change control, configuration management and other systems management processes.

Development of the Operational Model begins at the logical level and progresses to the physical level. The following points provide additional information on this progression. The logical level describes the characteristics and capabilities of the operational aspect of the system architecture in a technology independent and product neutral manner. This physical level describes characteristics and capabilities of the operational aspect of the system architecture in a technology and product dependent manner. Both levels may be further classified with respect to the specificity of sizing. For simpler designs, the model may progress from logical to physical within a very short timeframe. Progression toward the physical level will be accelerated by physical constraints placed on the design by the customer. At any given time, it is normal to have various aspects of the operational model at different levels. The reason is that various aspects of the design progress at different rates. For example, during discussion, the client may say that a certain component is constrained by standard to a specific product such as MQ. While the overall model may be primarily at the logical level, the nodes where MQ is likely to be installed have moved to a physical level. The point is to gather information where and when it is available and reflect that in the model.

Steps
Identify locations and/or layers of the system
Identify all locations and/or for the system. A location is a specific geographic location at which one or more nodes will be located. A layer may relate to an organizational aspect such as branch office.
Identify Nodes

Identify candidate nodes of the system. A node is an abstraction that contains a collection of functional components.. .

Assign Nodes to locations

Assign nodes to layers or locations as appropriate. A particular type of node may be assigned to multiple locations.

Identify functional Components and determine how they will be deployed
  • Identify all functional components
  • Identify Deployment Units, associated Components and map DUs to Nodes 
  • Determine node characteristics and specify appropriate hardware platforms.
  • Identify functional components required for technical infrastructure and middleware and specify appropriate software products.
Identify Non-Functional Requirements

Identify those non-functional requirements for each node. If Deployment Units are used, node requirements can be derived from the DU's that have been assigned to a node.

Identify inter-node connections
Identify inter-node connections necessary to support interactions between components assigned to the nodes.
Create Node descriptions
Create node descriptions that include information from the previous steps.
Describe System Operation
Develop walkthroughs using key use cases that illustrate intended function of the IT system.
Key Considerations

Cloud Computing has introduced significant change in the definition and automation of IT infrastructure. Your operational model should reflect that you have considered Cloud options and the new tools for software defined specifications of compute, storage and network resources

Enterprise Architecture Influence: Existing enterprise architecture is an important influence on solution design. For additional guidance on this topic, see the associated Guideline: The Role of Enterprise Architecture in Solution Design. The following EA artifacts (or their equivalents) should be considered when performing this task.

  • Business Roles and Locations (ARC 303) Provides a framework for the placement / deployment of applications, data & technology for use in development of the Operational Model.

  • Data Stores (ARC 306) Provides information that is helpful in defining enterprise-wide rules, polices and guidelines on data access, sharing, placement and management.

  • Enterprise Technology Framework (ARC 308) The ETF provides guidance in selection of components based on pre-established architectural building blocks (ABBs).
