Task: Document Architectural Decisions
The purpose of this task is to identify and document important architectural decisions where alternatives exist, choices are unclear and impact is likely significant.

Main Description

Various architectural decisions will be made regarding the proposed systems. These can be fairly general architectural principles or policies. They can also be fairly specific decisions regarding elements of the architecture. These may include prior decisions which will constrain this project in some way. The resulting work product will be updated during development of the architecture overview.

Ensure the issue or problem is clearly stated, evaluate the options that are available, make the decision and provide the rationale behind the decision, document the decision using the appropriate template.

Steps
State Issue or Problem
Clearly state what the problem or issue is that has to be addressed.  Obtain agreement from the appropriate stakeholders that this is the correct statement of the problem.
Consider Alternatives

 Build a list of alternative solutions to the problem or issue.  Review or consider one or more of the following when drawing up a list of alternatives:

  • Project objectives, goals, critical success factors, business drivers etc that may have been provided by the stakeholders
  • The system requirements (business processes, use cases, qualities, constraints and change cases).
  • Architectural principles, policies and guidelines drawn up by the stakeholder.
  • Relevant reference architectures, architectural patterns or assets.
  • Product literature available from vendors or their web sites.
  • Technical information published in books, journals, web sites etc.
  • Project documentation such as plans, key milestones and dates etc.
Make Decisions and Provide Rationale

Several factors need to be considered when making an architectural decision.  Consider the following questions as part of the decision making process:

  • Does the solution need to comply with a reference architecture or architectural pattern?
  • Are there any over arching principle, policies, guidelines or standards that need to be complied with or adhered to?
  • What systems management and support implications does the decision have?
  • What are the security and privacy implications of the decision?
  • How does the decision affect the total cost of ownership?
  • If the decision requires the use of vendor products, packages, consulting or support what implications does this have? For example what is the experience of the vendor?
  • Are there suitable service level agreements in place? Are there any liability or legal issues to address?
  • If the decision requires the use of a product or package what is the lead time on procuring these?  Is that consistent with project timescales?
  • Are there any change cases that have been identified that should be taken into account?

Once the decision has been made ensure the rationale for making the decisions is properly and clearly stated.  Some decisions can have far reaching implications and must be reviewed with all the key stakeholders before being finalized.

Document Decision
Document the decision together with the options considered and rationale used in making the decision.  Provide additional information (e.g. implications on other areas of the architecture, requirements impacted etc) as necessary.
Key Considerations

In a pre-sale context, the Architectural Decisions (AD) work product should document key decisions that will constrain the system design or selection of products and technology during implementation. There may be relatively few decisions in a pre-sale environment, but there will likely be additional decisions added during implementation. The AD work product is helpful in all but the simplest situations.  For guidance and examples of this work product, refer to the supporting materials section.

Ensure that that you fully document in the relevant Architecture Decisions the selection, constraints and considerations when using Cloud-based infrastructure, platform or application services. Cloud-based procurement changes the business model for IT solution components, removing the barrier for capital investment but not the requirement for the Technical Solution Architect to capture the key decisions and understand where the solution can scale and functionally evolve but also where the decisions will have future implications.

Although not officially part of this work product, architecture principles can be very helpful in considering architectural decisions (see note below). The client should be able to tell you if there are important IT or design principles that you should consider in developing a solution in the pre-sale environment. You may choose to develop other principles in collaboration with the client. The following table gives an example of how to record such principles.

Architecture Principles
Category / topic Complexity hiding
Principle / policy The programming model for client developers will hide the complexity of interfacing to legacy systems
Explanation In this context, client developers are responsible for developing applets, servlets and HTML. Legacy data will be accessed from within servlets. The servlet developer must be presented a common interface that is independent of the number and type of backend legacy systems.
Relevant requirements see non-functional requirement #xxx
Motivation Over time, the legacy systems will change and new ones may be added. It is desirable to insulate client programs from these changes. It is also desirable to simplify the client programming model by having a single interface for programmers to learn that does not change as backend systems change.
Implications 1. An integration server node will be required. 2. A client programming model must be selected. 3. An additional layer will exist that must be accounted for in performance modeling. 4. A new common business interface must be designed. 5. Consideration must be given in the operational model for global signon, and flowing of security and transactional contexts.
