Concept: Reuse
According to the Reuse Adoption Guidebook produced by the Software Productivity Consortium, reuse is defined as: “the use of an asset in the solution of different problems or different versions of a problem.”

Main Description

Assets in this case range from code level and architectural entities (classes, frameworks, architectures) to models (industry, analysis, design) to test cases, to experience and expertise. The ability to reuse assets in different problems or different versions of a problem is dependent on four key aspects of reuse. These aspects are:

  • Scope and context
  • Intention
  • Commonality
  • Variability analysis
  • Cost-benefit analysis

The following steps may occur in every phase of a project where asset reuse is feasible:

  • Requirements Gathering
  • Search for Assets
  • Understand Assets
  • Investigate Assets
  • Select assets
  • Adapt Assets
  • Reporting

Where higher-level models have been used, some of these steps may not be necessary in subsequent phases. For example, if an industry model were selected, then it would not be necessary to do searching and selecting at analysis time. In this case, requirements gathering would consist of transferring to the current phase the part of the model that is relevant.

Reusing assets makes sense only if it is less costly than creating the same capabilities from scratch. The cost of reusing/adapting assets must be compared to the cost of creation from scratch as part of the selection step. The cost of using the asset includes the cost to find it, understand it, and adapt it. Requirements gathering should be done so as to give an indication of the amount of adaptation necessary. Experts on the particular assets should be available to help determine these costs.
