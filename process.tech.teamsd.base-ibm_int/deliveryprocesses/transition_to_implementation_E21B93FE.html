Task: Transition to Implementation
This task is to ensure that promises during pre-sale solution design are understood and can be implemented by delivery.
Purpose
To make a smooth transition to implementation, delivery, deployment.

Main Description

Transitioning from the end of the pre-sale cycle in the Develop activity to the start of the post-sale cycle in the Implement activity results in a change in project leadership and the engagement of a set of delivery resources. It is critical that this hand-off be efficient. In a traditional solution development process this transition to implementation should include some sort of Project Kickoff.  The Project Kickoff meeting should be held to educate the team on the client, project scope, timetable, deliverables, etc.  The Project Plan will become much more specific as implementation begins.

In a more agile and collaborative solution development process, initial implementation may have been started during the Develop activity via a Proof of Concept (PoC). Such a PoC may even have resulted in the development of a Minimal Viable Product.

Whatever the delivery approach, the Technical Solution Architect is responsible for ensuring that promises made are understood and can be implemented by delivery.

The delivery resources might be IBM Services, the client, a Systems Integrator, or some combination of these.  Leadership of the project will move to the implementation/delivery architect (although this role might be performed by the same person as the Technical Solution Architect Role). This task gives guidance for a transition to IBM Global Business Services but a similar approach is needed no matter who will be implementing and delivering the solution.

If the size and complexity warrant it (high Business Transaction Type (BTT)), then this is the time for the initial Project Management Review (PMR). If the project implementation has been broken into phases, a PMR at the end of each phase is appropriate. Even if the project isn't being broken into timeboxed or functional "chunks", the project might include phases such as Solution Build, Solution Test, Pilot, Deployment, Run (Production).  Project Management Reviews are a perfect opportunity for the Technical Solution Architect to understand how the proposed solution is actually being implemented. 

As a project progresses, there will be further elaboration of the work products, typically documented as Architectural Decisions.  There may also be changes to requirements. These changes can result from the discovery of previously overlooked requirements or changes in scope that require significant new function. Such changes should be carefully documented and managed because they may have significant impact on cost, schedule and client value. These changes may be documented as Change Cases (ARC 118).  In almost all cases, there will be lessons learned and best practices to be harvested even if only as a learning experience for the Technical Solution Architect. 

Key Considerations

Like other tasks in the Team Solution Design Activity: IMPLEMENT Client Solution, this task is led by others in IBM and the client. The role of the Technical Solution Architect is to ensure promises made to the client are fulfilled and that lessons learned are acted upon. The only way to make this happen is for the solution designer to have at least an advisory role in Implementation.

If there is a teamroom or community used to store client information, the results of this task will be stored there for use on later projects.
