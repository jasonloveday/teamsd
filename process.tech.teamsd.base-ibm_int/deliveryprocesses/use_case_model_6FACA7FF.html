Work Product (Artifact): Use-Case Model (ART 0508)
This artifact captures a model of the intended functions and environment of the system and serves as a contract between the customer and the team.
Purpose

This artifact presents an overview of the intended behavior of the system. It is the basis for agreement between stakeholders and the project team in regards to the intended functionality of the system. It also guides various tasks in the software development lifecycle.

Note for pre-sale solution design (Team Solution Design):  In a pre-sale context this artifact typically serves to:

  • Establish a small number of important scenarios that depict how the user will use the proposed system.
  • Provide a basis for basic inferences regarding functionality that must exist within the proposed system and its user interfaces.
  • Identify functional requirements that more are elaborate than those described in the Project Definition
  • Provide a basis for planning a proof of concept
  • Provide input to the Component Model
  • Provide a basis for high level walkthroughs using the architecture overview diagram as a means of developing and explaining the architecture

Key Considerations

Note for pre-sale solution design (Team Solution Design users):

In a pre-sale context, this artifact will contain a small number of use cases (3-4 might be typical) that describe some of the most important functional requirements of the proposed system. The model uses graphical symbols and text to specify how users in specific roles will use the system (i.e., use cases). The textual descriptions describing the use cases are from a user (or external IT system) point of view; they do not describe the system's internal structure or mechanisms.

The use case model is described by the following constructs. Note: Only those shown in bold type are typically required in a pre-sale context. If additional detail is required, regarding other parts of the model, refer to the complete guidance associated with this work product.

  • Actors: These typically represent users performing in some role such as registered user.
  • Use cases (includes basic scenario definition)
  • Associations between actors and uses cases 
  • Use Case View this may not be necessary in a pre-sale context. 
  • Problem domain concept definitions
  • Flow of events table
  • System sequence diagram

Actor names, actor descriptions,  use case names, use case business events, use case overviews as well as associations between the actors and the use cases provide an overview of the functional requirements. The other constructs of the model may only be required in more complex situations.

For a discussion on the necessity of this work product and appropriate levels of elaboration for a pre-sale environment, consult the guidance entitled Pre-sale Work Product Usage for Team Solution Design. 

Enterprise Architecture Influence: Existing enterprise architecture is an important influence on solution design. See the associated Guideline: The Role of Enterprise Architecture in Solution Design for guidance on EA artifacts (or their equivalents) that should be considered when developing this work product.

Tailoring
Impact of not havingWithout this artifact, it is difficult to determine all of the relationships between actors and use cases. It is also difficult to gain an understanding of how different use cases relate.
Reasons for not needing

You might not need this requirement if the overall set of use cases is small and has no complex relationships.

You also might not need a use-case model if you use an alternative approach to document functional requirements.

Representation Options

Representation options include reports and diagrams from UML modeling tools, graphical representations created by using drawing tools, and drawings on whiteboards. Most of the information in the use-case model is captured in the use-case specifications.
