Work Product (Artifact): Connection (ART 0572)
This artifact describes a connection or a structured collection of connections between nodes within the operational aspect of the architecture of an IT system. Connections represent, at an architecture level, the networks between the IT system’s infrastructure platforms.
Purpose

The purpose of this artifact is to provide a structured description of the connections between the nodes within the operational aspect of the architecture of an IT system.

The purpose of this artifact depends on the context in which it is developed. For example,

  • In the context of the design and development of an IT system, this artifact is developed as part of the overall operational architecture aspect to describe the connections between nodes and provide an input to the design of the network required to support the desired service level characteristics.
  • In the context of a reference or enterprise architecture, this artifact provides a pattern that can be used as a basis for analyzing an existing IT system or as a basis for architecting a new IT system.

Description
Main Description

A connection provides connectivity between two nodes to support the interactions between components placed on those nodes.

This artifact’s (and that of its contained artifacts) terminology is defined within the System Description Standard R3 Semantic Specification (see Guidance for link) which includes definitions for:

  • Components - modular units of functionality, which make this functionality available through an interface.
  • Deployment Units – a grouping of facets of a component that have similar characteristics.
  • Interactions – identify the messages exchanged between one or two resource containers in the context of a collaboration, and the sequencing of these messages via their associated send/receive events.
  • Nodes - a collection of components fulfilling a specific responsibility with a certain quality of service within the target system.
  • Connections - supports the required connectivity between connectable model elements. 

Depending on the development approach adopted and/or the context, it can be very helpful to create levels of this artifact.  The two basic levels of this artifact are

  • Logical:  This level describes the characteristics and capabilities of a connection in a technology independent and product-neutral manner.
  • Physical:  This level describes characteristics and capabilities of a connection in a technology and product dependent manner.

Both levels of this artifact may be further described at three different sub-levels relating to the degree to which the level has been sized:

  • Unsized (or unranged) – the technology and products are specified to support the components deployed but not sized and without detailing how the system will achieve the service level characteristics.
  • Ranged – the technology and products are specified to support a bounded range of requirements for a defined range of circumstances.
  • Sized – the technology and products are specified and sized to support the components deployed and details how the system will achieve the service level characteristics.
Brief Outline

This artifact contains:

  • A list of connections and/or connection groupings
  • For each connection
    • Connection name
    • A description of the connections between nodes (or a description of a structured collection of connections),
    • Connection characteristics and capabilities,
    • Nodes which the connections provide connectivity for.
Notation

This artifact is documented in a combination of structured text, structured tables, matrices and diagrams using a specialized notation and naming standards which are defined in the IBM System Description Standard V3 and the IBM Architecture Description Standard (ADS) Operational Model Notation Guidance V2.   Please refer to these for details of the notation.

Tailoring
Impact of not having

Without this artifact, there is no specification of the IT system’s physical infrastructure.

Reasons for not needing

This artifact may be unnecessary if the system is very trivial and either consists of only a few nodes when it may be possible to jump straight to the design of a network.
