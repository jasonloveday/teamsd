Work Product (Artifact): Project Definition (ENG 343)
The document that describes the shape of the project and includes: the objectives and scope, the stakeholders, the proposed organization with responsibilities, and the major risks associated with the project.

Note for pre-sale solution design (Team Solution Design users):  This version of Project Definition is intended only for pre-sale use! The purpose of this artifact is to provide a common understanding of the project by each member of the team.  By establishing a brief answer to the questions: what, why, when, where, how and who; this artifact gives the team a context within which to work and provides a concrete starting point for solution design.  Note that Project Charter (ENG 341) is typically not used and not available for pre-sale solution design.

Description
Main Description

Note for pre-sale solution design (Team Solution Design users): The Project Definition work product answers the following questions:

  • What are the business problems we're trying to address?  Why is the client doing this project?
  • What are the client's success criteria? What value is the client expecting from this project?
  • When are we going to get it done (by phase)?
  • Where will the solution be deployed (by phase)?
  • How will the project get done?
  • What are the key functions needed that will determine our solution?
  • Who are the key participants (customer, IBM, vendor)?  Who is sponsoring (accountable) for this project?  Who is Responsible, Consulted or Informed? 

The planning effort often identifies much of the information needed to turn the validated opportunity into a well defined project.  The Opportunity Plan, if it exists, may contain the following information and might need to be updated: Client team including person accountable, client's expectations from the project, opportunity assessment including "is it worth it to pursue this opportunity?".

Key Considerations

While the project manager is generally responsible for the creation and maintenance of the Project Definition document, input for the document comes from many sources and from most team members in a leadership position on the project. Once complete, it is approved according to the document control procedure for the delivery organization

Note for pre-sale solution design (Team Solution Design users): This version of Project Definition is intended only for pre-sale use!
The project definition is expected by Team Solution Design and is required for every project since most other work products used in Team Solution Design are dependent on it.  Together with architectural decisions, it ties the other work products together and communicates the project status to the client and other team members.  The project definition and viability assessment are updated throughout the life of the project. This is why project definition and viability assessment are the primary mandatory work products for reviews with the client and IBM teams.  Of course, which parts of the project definition and viability assessment you share with the client depends on your relationship.  
There are two templates associated with this work product. In a pre-sales context it is normally recommended that you select the Project Definition Template for Team Solution Design. This template is based on the standard template, but tailored for Team Solution Design pre-sale use.  Some sections are not normally required in a pre-sale context and are indicated as such.

There is also a Guideline (in More Information) which includes Project Scope, Buying Behavior and RACI (an approach to capturing the level of responsibility as Responsible, Accountable, Consulted or Informed). In cases of IBM implementation, the project definition should be be conveyed to the implementation team for further elaboration.

For a discussion on the necessity of this work product and appropriate levels of elaboration for a pre-sale environment, consult the guideline entitled Pre-sale Work Product Usage for Team Solution Design.






Tailoring
Impact of not having

Lack of this artifact leads to:

  • Possible misunderstanding of the projectâ€™s objectives by the delivery organization.
  • Unclear view of the feasibility of the solution which can affect the reliability of the plan being defined.
  • Difficulty in ensuring there is a common view of the solution and of the related approach inside the project team and with the sponsor.
  • Lack of clear boundaries for the project scope that leads to:
    • Risks of gaps or overlap between the roles acting on the project.
    • Poorly defined or inadequate plan for the project.
    • Subsequent difficulties in controlling the project.
  • In the shorter term, difficulty in controlling project planning activities.
Reasons for not needing

This artifact, or another document with similar content is needed on every project. On a small project, it can be a short document.

Note for pre-sale solution design (Team Solution Design users): Although this artifact is required for all projects, it might be simplified for simple projects (like Value for the Money opportunities).  For example, a formal section to capture functional requirements might not be needed.
