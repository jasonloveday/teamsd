Sample Node Description


Node Name Network Intrusion Detection Node
Node Description This node serves to monitor the DMZ for unusual traffic patterns that might indicate an attack on the network.  Ideally this node will detect an intrusion and notify the system/network manager that intrusion has occurred.  Note that an attack is defined as unauthorized attempts to gain access through the public firewall.  Intrusion is when such an attack has successfully penetrated the public firewall.  When an intrusion occurs and is detected and notified, it is red alert situation. Intrusion can come via hacks through the back door, hacks from staff from same or remote location or from successful hacks through the firewall. Detection has to be done with a very high degree of accuracy and on a timely manner (preferably within minutes).
Presentation function Monitor, keyboard, mouse and remote display functions
Processing function

Data Local stores for audit logs
Infrastructure Use of architectural conventions for Intrusion Detection Modules, Messaging specification for audit data and its transmission or some of the approaches that have to be followed.  These are components of the Common Intrusion Detection Framework.
Presentation services

Processing services

Intrusion detection is typically based on analyzing network auditable events such as

  • Users logging in at unusual hours
  • Unexplained reboots
  • Unexplained time changes
  • Unusual error messages
  • Failed login attempts
  • Users logging in from unusual sites / locations

This node should support the ability to track and analyze such network events on a real-time basis.

Sample general walkthrough description for Use Case "Obtain Customer Relationships"


For a visual summary of the walkthrough please refer to the event trace in the figure above.

  1. The scenario begins as the customer will enter the URL of the logon page of the e-business application in the URL field of the web browser on the Client Node and press the Enter key.
  2. The URL request will be forwarded to the e-business system across the Internet.
  3. The request enters the system and is routed by the Edge Server Node to the Reverse Proxy Node that can service the request. 
  4. The Reverse Proxy Node determines that the request is for the logon page which is a Static HTML form and forwards the request to the Web Portal Node.
  5. The Web Portal Node will retrieve the static form and determines that the form needs to be encrypted.  The form is encrypted and is returned to the web browser.
  6. The user will enter his User ID and password into the secure HTML form and press the Submit button in order to send this information to the system for authentication. 
  7. The form is sent back to the e-business application and is sent to the Reverse Proxy Node.
  8. The Reverse Proxy Node determines that this form needs to call a Java Servlet for authenticating the user and it forwards the request to the Web Application Server Node.
  9. The Web Application Server Node will call the Login Servlet.  This servlet issues a Login request to the Directory and Security Node through a JNDI call. 
  10. Once the user ID and password are authenticated, the Web Application Server node uses the retrieved user authorizations to build the session context for the user.  It will then issue a call to the Integration Hub Node to retrieve the user’s account information from the backend enterprise system.
  11. The Integration Hub Node will format the request and issue a call to the Enterprise System Node.
  12. The Enterprise System Node will execute the program that retrieves the user’s data and send this response back to the Integration Hub.
  13. The Integration Hub will reformat the response and send it back to the Web Application Server Node.
  14. The Web Application Server will call the appropriate template JavaServer Pages (JSP) to format the response data as a HTML file.
  15. The HTML response file is encrypted and sent back to the Web Browser.
