Main Description
The following are some of the general requirements from the ESS e-business Reference Architecture that are very useful as examples to keep in mind not only while architecting a solution, but also when gathering requirements and setting customer expectations. These examples are based on experience gained from deploying many e-business solutions.They are examples that communicate various aspects of an e-business solution. However, note that the specific needs of any given application are likely to be different and therefore must be analyzed and researched to a sufficient degree. Also, these requirements would likely be split into several separate and more specific requirements with more specific names.
 
An additional example developed using the SA4TeamSD configuration of IBM's Rational System Architect is included at the end of this guidance.

Index Requirement Name Priority  (MoSCoW) Value(s)
  NFR001 Connection Response Time Must Normally, we would expect a requirement to be <5 secs. But Web technology can make this a challenge, so be careful before agreeing to a figure. For external customers (Web connected) establishing a connection may be > 30 sec. For internal users, normal connection times should be obtainable.
  NFR002 Read individual on-line response time for business transactions Should

The expected requirement is 1-2 seconds average. Take care when setting requirements for Web access. Response time through the Internet is highly unpredictable. Sub-second response is difficult to achieve with most thin-client topologies.

  NFR003 Read set on-line response time for business transactions Should See comment above
  NFR003 Update individual on-line response time for business transactions Should See comment above
  NFR004 Online transaction processing rate Should Expected value is up to 50/second. Need ability to scale to large servers or to spread load across many servers dynamically. Numbers in excess of 50/sec require special consideration.
  NFR005 Number of online users Must This number is not relevant for users connected through the Internet, which employs a connectionless protocol. BUT - it can become very relevant in that it can provide very high peak volumes in the next row. In a non-Internet environment, a typical large Enterprise requirement may be upwards of 20,000 online users.
  NFR006 Number of concurrent online Users Must In a non-Internet environment, a typical large Enterprise requirement may be upwards of 20,000 online users.
  NFR007 Batch processing volumes Should Expected values for Enterprise customers are in the range 20K-200K per day, in terms of Bills, Reminders etc.
  NFR008 Server Database Size Should For an Enterprise, will typically exceed 100GB for main customer files (e.g., the persistent storage requirement for a Utilities Customer Service System may be around 300GB for say 3 Million customers.) and 1TB is possible for certain Enterprises. These numbers appear to be increasing rapidly as more and more data is kept about either customers or their activities (meter readings etc).
  NFR009 End-user availability Must Typically for an enterprise, 23*7 is demanded. Question the requirement if 24*7 is demanded as this can pose technical challenges, e.g. in Database housekeeping. Internet connection may mandate very high availability due to the need to provide "all hours" worldwide service.
  NFR010 Distribution of Systems Management Should If the applications are required to be available 24*7, when do necessary systems management functions such as database tuning, upgrades, application of software patches, get completed? In other words, systems management functions need to be distributed around the clock and have to be performed at the "right" times. The times that are "right" for the application(s) are determined based on the availability, and usage patterns of these applications.
  NFR011 Strict Security requirements TBD Be careful with architectures that have strict security requirements (e.g. Health Insurance Portability and Accountability Act (HIPPA)).  Current software and/or hardware solutions provide limited support for these types of requirements, particularly when they require integrated end to end support.  If extensive application involvement is needed to support the security requirements, that in itself may be an issue.  Solutions where the application plays an active role in authentication, authorization, and Non-Repudiation are generally considered less secure than solutions where the security is implemented through the infrastructure.
  NFR012 Application integrity Must Expected to be Critical for this Context.
  NFR013 Access Granularity Should Financial institutions may require field level security, in other words, certain users may not be permitted to view the content of certain fields (the classic example is the managing director's salary).
  NFR014 Type of Threat Must Internet connection should assume that there will be deliberate attempts to compromise security
  NFR015 Single Login Could Very difficult to achieve with existing products.
  NFR016 Trusted Environment Must Frequently, Enterprises consider Client tiers inherently untrustworthy.
  NFR016 Audit Trail Would Not Increasingly demanded, and not well supported by current DBMSs.  Will likely involve additional application coding and risk.
  NFR017 Strategic versus Tactical TBD The architectures recommended by ESS as typically focused on more Strategic solution: may be over-engineered for Quick & Dirty ones.
  NFR017 Adaptability/ extensibility Could Increasingly important and has a high impact on application design.
  NFR017 Frequency of changes TBD Typically frequent.
  NFR018 Platform Neutrality Should Often demanded.  Helpful in any event to provide insulation from specific technologies.  ESS provides logical architectures to decouple early parts of the design from physical choice.
  NFR019 Process change TBD Especially important in Insurance companies, which have a strong workflow need.
  NFR020 Rules based Must Really, a solution to the flexibility problem rather than a requirement in its own right. SAP provides an example where in order to provide "Replicable Solutions", 10.000 tables need to be customized.
  NFR021 De jute and de facto standards Must Conformance to Internet protocols and standards is critical
  NFR022 Importance of batch processing TBD Usually critical in the Enterprise set we are considering.
  NFR023 Batch/Online Concurrence Could Increasingly demanded as a way of addressing the availability issue.  Can be done but requires careful design of the solution, especially data locking strategies.
  NFR024 Importance of legacy integration Must Critical for this Context.
  NFR025 Style of access TBD Typically all may be relevant and needed.
  NFR026 Data updates TBD If multiple legacy systems need to be handled inside one mid-tier transaction, severe problems are posed.
  NFR027 Distribution of printing TBD Customers who produce large volumes of printed output, such as bills, catalogs, newsletter/communications, etc. will benefit from carefully planned distribution of such printing activities.  Geographical distribution to optimize on time as well as sorted & batched printing capabilities to optimize on postal costs should be taken into consideration.
  NFR028 Image TBD ESS provides some Image models and architectures, but they are not yet fully integrated into the ESS Transactional Architectures.  Image Processing is very important for some large Insurance companies.
  NFR029 Size of Development TBD ESS constructs are intended to support very large development projects.
  NFR030 Cost Must The particular stress which customer puts on costs could have a dramatic effect on design.  A "Strategic" solution may be permitted sophisticated rules-based architecture in order to minimize the costs of change in use.
  NFR031 Distribution of Users Must All except Mobile are included (for these see Pervasive Computing Reference Architecture)
  NFR032 Type of employment TBD Affects the UI requirement, and the sophistication of HELP text.
  NFR033 User Interface TBD Current Web technology (Java) cannot support Advanced OOUI.  ESS thin-client architectures will mitigate against very complex UIs.  If Client insists on a high-function GUI, fat clients may be needed.
  NFR034 Multiple time zones TBD If this is critical, there will be implications for availability
  NFR035 Maturity Should If client demands maturity and if volumes are high, the Transactional ORBs should be approached with caution at this stage.
  NFR036 Depth of Object Orientation Should ESS supports both OO and Object-based development.

The table below contains a sample set of questions that may be posed to the clients to solicit responses regarding the non-functional requirements.
Question Response Explanation
Number of Loans Processed 20,000 Revenue model estimates 10,000 loans processed in the year 1.  All these loans are processed in the 4 month peak period
Total number of anticipated users of the system 120 The number of anticipated users of the system is a fraction of the overall population of end users.  How an organization would promote the usage of an e-business applications would determine what percentage of the overall population of end users would use the system. 
Anticipated number of Concurrent Users 50 This value was based upon 20000 applications over a 4 month time = 5000 applications per month.  5000 application over 30 days = 166 applications per day.  Assuming a scenario in which all applications are created in 3-4 hours window, the anticipated number of concurrent users is approximately 50
Expected Availability 12 hrs / 6 days per week. Availability is best described in terms of XX% Uptime instead of 24X7.  24X7 could mean 100% availability and such requirements result in extremely expensive infrastructure.
Expected Response Time 30 sec The response time could vary substantially between static and dynamic pages.  In the case of dynamic pages involving backend data processing the response time would depend on response time provided by the backend systems.

 
Question Response Explanation
     
Number of Loans Processed 20000  
Number of Logical Transactions or Activities Performed by the users for a single loan application 6  
Logical transactions processed in a peak hour 300 20000 * 6 logical transactions are processed in a 4-month period.  Assuming majority of the loans are processed in 4 hours out of the 12 hour days = 20,000 * 6 / (4 * 30 Days * 4 Hours) 
Average number of Page Requests per Logical Transaction or Activity 20 12 for submitting the loan application; there is connectivity to the FICO for credit scoring; TO accommodate these Credit ASP transactions and for SSL connectivity that is required by both Seller desk to and Credit ASP server to server communication we have increased the number of page requests per logical transaction to 20.
Page Request Per Peak Hour 6000  
Average Number of Hits per Page Request 10 A hit is a request for a file (not a web page) to a web server. A file could contain text, image, audio, video, binaries, etc. A web page is composed of one or more files. On average, a web page contains ten files. In other words, a hit on a page could result into requests for ten files.  The variance on this value is quite large. One of the world’s busiest web sites sees more than twelve million hits during its peak hour.  In the case of Internet Applications the number of users could depend on how the site is advertised. In the case of an Intranet Application the number of users could depend on what are the alternative methods available to accomplish the same task and how the employees are motivated to use the Intranet Application.  
What is the Peak Hit Rate per Hour (PHRH) 60000 PHRH = Page Requests per Peak Hour * Average number of Hits per Page Request (Assuming 1500 Loans Per Store Per Month * 600 Stores = 900,000 Loans Per Month). Each Loan Could have 10 distinct Logical Transactions. => 9,000,000 Logical Transactions Per Month via Web Interface. They operate 220 Hours Per Month => Logical Transaction Via Web Per Hour = 9,000,000 / 220 = 40,909 Each Logical Transaction is composed of 2.5 Page request on an average, => 2.5 Page Requests X 10 = 25 Hits => 25 Hits X 40,909 = 1,000,000 Hits on an Average Per Hour
What percentage of Pages is Dynamically generated? 75% Since loans are static pages
What is the processing complexity of the dynamically Generated Pages? 100 Rendering dynamic content to a browser requires significantly more processing power than does serving up static content from a file system. The extent of additional resources required varies with the method employed. A simple approach to estimating resource consumption for dynamic pages: If a static page takes "X" resources, then the dynamic processing, database look-up, page formatting, etc. will require, "N" times "X".  This assumes the workload resource consumption profile (for example the ratio of disk I/O to LAN bandwidth) for the two types of processing are identical with just a multiplication factor required.  For example, if a total of one static page and one dynamic page are served per hour, and the dynamic page consumes 100 times, the resources of the static page then one can think of serving those two pages as the equivalent of serving a total of 101 static pages.
What's the average static file size (Kbytes)? 30 KB Logo image is 47KB to 68KB, HTML is 6KB to 800KB, GIF is 8KB to 1GB, JPEG is 37KB to 800KB, etc.
How much disk space will be required for the web server's content? 10 GB The Web Server may not host most of the content instead rely on Distributed File Systems to Host the Content.  In that case the majority of the hard disk space needed would be on the File Server
Average number of Backend Transactions Per Page Request 5 JDBC Calls    
What is the percentage of logic between Web App Server and Legacy Applications? 100% on web App Server  
Concurrent Users hitting the backend Systems at Peak Load 20 Connections    
What is the LAN bandwidth 100 Mbits/sec Universal Server Farm (USF)
What is the internet connection bandwidth (Kbits/sec) at Sellers desk top 28.8 kbs Modems: 28.8 / 56 Kbits/sec, ISDN: 128 Kbits/sec, T1: 1,544 Kbits/sec, T3: 45000 Kbits/sec, ATM: 155000 / 622000 Kbits/sec.
