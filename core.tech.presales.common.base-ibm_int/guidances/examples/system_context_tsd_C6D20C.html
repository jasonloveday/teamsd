Main Description

The System Context may be developed as a free form diagram or as a more formal set of views. The free form System Context Diagram typically consists of the following major parts:

  • The proposed (target) system (normally simplified by illustrating it as a single element in the form of a circle or rectangle.)  The result is a  visible boundary between the proposed (target) system and relevant external systems.
  • Actors and external systems with which the proposed system must interoperate
  • Interfaces on the boundary which connect some aspect of the proposed (target) system with external systems. In the diagram below, these are suggested by the arcs connecting the external systems to the target system.
  • Information regarding events, formats, protocols, etc. of the boundary interfaces

The elements within the target system boundary are those over which the development team and the client have some control. The users and systems outside the boundary of the target system are those which affect the system operation and development but are beyond the direct control of the developers and client. Changes may be required in these systems.

In more complex situations, it may be helpful to develop more formal views of the system context. These views are based on four viewpoints which reflect various combinations of Functional, Operational, physical and logical. The possible views include:

  • Logical Functional view: Reasoning about what the system does for or to its actors
  • Physical Functional view: Reasoning about the users and IT systems it connects to
  • Logical Operational view: Reasoning about the nature of the access mechanisms used by its actors
  • Physical Operational view: Reasoning about the connections needed with its users and other IT systems

The following example illustrates the logical functional view. The lines connecting the target system with actors and external systems depict logical "Exchanges". An exchange represents the collection of all interactions across all collaborations between the actor or external system and the target system. In an operational view, these lines represent the access mechanism which connects the actor or external system with the target system and supports the exchange. When creating these formal views, it will be helpful to use a design tool such as the SA4TeamSD configuration of IBM's Rational System Architect. Because of the additional detail in these formal view, it will be more practical to keep the annotation for the elements separate from the view. For more information on developing these formal views, refer to the associated guidance.
