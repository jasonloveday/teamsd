Main Description

The following represent examples of architectural decisions. Note that in a pre-sales context, not every row of the Architectural Decisions template is needed for every decision.  Also, this work product is very helpful in capturing most decisions that impact the solution design, not just those that impact the architecture.

Architectural Decisions
Subject Area Programming model
Architectural Decision Define the level of integration which will be exposed to the client application programmer (including servlets)
Issue or Problem The issue is how much detail regarding the backend data sources will be exposed to the servlet developers.
Assumptions Backend data sources will be two or more CICS legacy systems EJB support will be available as an option
Motivation This decision will affect the number of people who must be skilled on the complexity of backend system interfacing
Alternatives 1. Expose API's associated with CICS gateway 2. Develop business object interface that will hide source of data
Decision A business object interface will be developed that will hide the CICS access
Justification 1. This will restrict more complex knowledge of integrating with CICS systems to a small team instead of all developers. 2. If changes occur in the backend systems, application logic is protected, with changes limited to data access methods of the business object layer. 3. This supports the architectural principle on Complexity hiding.
Implications 1. A formal design must be accomplished for business object layer
Derived requirements 1. EJB and related tool support may be required 2. Choice of EJB access (if used) must be made and training for this must be made available.
Related decisions CICS access software must be selected

Architectural Decision
Subject Area Enterprise Connectivity
Architectural Decision Means to communicate between with backend systems and databases
Problem Statement The e-business application should integrate with several backend systems and databases.  The question is how to best communicate with these systems.
Assumptions Interactions with back-end systems will be a mix of synchronous and asynchronous communications.
Motivation Because of the degree of application integration needed, a decision must be made on enterprise connectivity to avoid inconsistent and unmanageable connectivity.
Alternatives 1. Point-to-Point Connectivity This alternative connects the application server connect directly and individually to each necessary enterprise system, using the protocol best suited for that system.  For example, connectivity to the mainframe could use Common Programming Interface for Communications (CPI-C).  Alternatively, messages could be exchanged individually with each system using a messaging product such as Message Queuing Series (MQSeries) or Electronic Data Interchange (EDI).  A very basic option would be to exchange data via File Transfer Protocol (FTP).

2. Integration Hub This alternative centralizes all enterprise communications at a common intermediary called an Integration Hub.  Business-level messages are given to the hub for delivery.  The hub routes the message to the correct enterprise system, translating the message if necessary into a format acceptable by the enterprise system and communicating with the enterprise system in an appropriate protocol.  This hub can be used to support synchronous and asynchronous integration.

3. Business Workflow This alternative adds onto the Integration Hub by adding business process functions to its capabilities.  A message would trigger a set of further messages to different destinations based on a pre-defined workflow for that business process.
Decision The e-business Reference Architecture will use the Integration Hub to enable enterprise connectivity. This alternative provides a consistent message based option for integration with disparate backend systems.  Further this option can be easily extended as needed to support Business Workflow as needed.
Justification Because of the number of different systems that must be accessed by any e-business application, it would seem prudent to have the application server communicate using one method (messages) to one system (the Integration Hub).  This also provides the flexibility of relocating data and functions to different enterprise systems with no change to systems that access that data.  Furthermore, the Integration Hub will simplify communications among all legacy systems and databases, allowing other projects within the enterprise to leverage this investment.

The translation capabilities of the Integration Hub are also a big advantage in the business-to-business world.  It is expected that this hub would be accessible to authorized trading partners, each of whom may adopt a different message “standard” for their business-to-business  (B2B) communications.  The Integration Hub would be able to translate these different message formats into a consistent format for communication with the organization’s enterprise system.

Another advantage of this option is its ability to support different collaboration models such as blackboard, point-to-point or Publish/Subscribe.

Although the business workflow option is intriguing and seems to offer benefits, at this time there does not seem to be a sufficient gain in this process for the cost incurred.  However, as the organization’s business and technical processes evolve, this option could be added onto the Integration Hub’s functionality.
Implications All enterprise applications will need to be enabled for working with the Integration hub.  A set of business messages will need to be defined, as well as translation rules for other systems.
Derived requirements
Related Decisions AD2, Enterprise Message Format


Architecture principles can be very helpful in considering architectural decisions. The client should be able to tell you if there are important IT or design principles that you should consider in developing a solution in the pre-sale environment. You may choose to develop other principles in collaboration with the client. The following table gives an example of how to record such principles.



Architecture Principles
Category / topic Complexity hiding
Principle / policy The programming model for client developers will hide the complexity of interfacing to legacy systems
Explanation In this context, client developers are responsible for developing applets, servlets and HTML. Legacy data will be accessed from within servlets. The servlet developer must be presented a common interface that is independent of the number and type of backend legacy systems.
Relevant requirements see non-functional requirement #xxx
Motivation Over time, the legacy systems will change and new ones may be added. It is desirable to insulate client programs from these changes. It is also desirable to simplify the client programming model by having a single interface for programmers to learn that does not change as backend systems change.
Implications 1. An integration server node will be required. 2. A client programming model must be selected. 3. An additional layer will exist that must be accounted for in performance modeling. 4. A new common business interface must be designed. 5. Consideration must be given in the operational model for global signon, and flowing of security and transactional contexts.
